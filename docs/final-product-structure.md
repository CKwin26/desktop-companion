# Final Product Structure

这份文档用来冻结一个问题：

**我们最终要做的，到底是什么。**

如果后续某个功能看起来很酷，但会把产品往“复杂任务管理器”“周报整理器”或“桌宠皮肤包”方向带偏，就以这份文档为准。

## 1. North Star

我们的最终成果不是：

- 一个披着桌宠皮肤的待办工具
- 一个周报/日报整理器
- 一个功能很多但存在感很弱的桌面小窗

我们的最终成果是：

**一个常驻桌面的、能理解多个任务主线的 RA。**

它应该像一个一直在你旁边的小人，但核心能力不是“陪你一下”，而是：

- 你点她，她会回你
- 你乱了，她会看出你其实在说哪几条不同主线
- 你说一句，她会决定要不要替你记住，以及记到哪条主线下面
- 你后面再追问，她知道你在延续哪条项目线
- 你分心太久，她会提醒你现在哪条线最该推进

所以：

- 第一入口是 `对话`
- 任务系统只是 `记忆层`
- 梳理只是 `帮助你理解和推进多条主线` 的手段

## 2. Product Definition

一句话定义：

**自然语言驱动的桌面 RA，能理解多个并行任务主线，记住每条线的关键信息，并帮助用户在不同主线之间切换和推进。**

用户感知到的核心体验应该是：

1. 她一直在桌面上
2. 她不是冷工具，而是有性格的角色
3. 她先聊天，再决定是否记忆
4. 她帮助你看清“我现在其实手上有哪些主线”
5. 她帮助你决定“下一步该推进哪一条”

## 3. Core Principles

### `对话优先`

用户面对她时，应该首先想到：

- “跟她说一句”

而不是：

- “打开面板填一个任务”

### `轻记忆，但必须能理解多主线`

她需要记住的不是完整项目系统，而是多条主线上的少量高价值信息：

- 当前主线
- 阻塞点
- 下一步动作
- 提醒节奏
- 主线之间的归属关系

### `陪伴感比报表感更重要`

她存在的价值不是生成总结文档，而是降低启动成本和拖延阻力。

### `桌面存在感必须成立`

不开大窗口时，她也应该成立。

也就是说：

- 小人本体要能独立存在
- 右侧一句气泡就能传递状态
- 不该强迫用户进入复杂界面

### `AI 是增强层，不是产品本体`

没有 AI 时，她也应该能工作。

有 AI 时，她会：

- 更会接话
- 更会梳理
- 更像一个角色

## 4. Final Layering

产品最终分成 5 层。

### `1. Pet Shell`

这是桌面上的小人本体。

它负责：

- 常驻桌面
- 动作和表情
- 点击、拖拽、吸边
- 小气泡
- 最小展开/收起

它不负责：

- 任务逻辑
- AI 决策
- 复杂数据处理

### `2. Conversation Core`

这是第一入口，也是人格层。

它负责：

- 用户输入
- 多轮对话
- 人格语气
- 情绪接住
- 追问与回应

它的目标不是“正确分类一条命令”，而是：

- 让用户觉得她真的在听

### `3. Task Memory`

这是轻量任务与项目主线记忆层。

它负责：

- 记住主线任务
- 记住阻塞点
- 记住下一步
- 记住提醒时间
- 在后台记住“这堆事项像哪条项目线”
- 在后台维持“你当前有哪些并行主线”

它不是：

- 完整看板
- 大而全项目系统
- 多层级任务树

它内部可以再分成两块：

- `Active Memory`
  - 当前主线 / 阻塞 / 下一步 / 提醒
- `Project Cognition`
  - 混合清单属于哪条项目线
  - 是旧项目还是新项目候选
  - 每条项目线最近挂着什么

重点是：

- `Project Cognition` 不是附属功能，而是核心能力
- 但它依然应该停留在“后台认知”，不是前台项目管理界面

### `4. Guidance Engine`

这是监督和重新启动引擎。

它负责：

- 什么时候提醒
- 什么时候追问
- 什么时候帮你梳理
- 什么时候别打扰

它的输出应该是：

- 一句轻推
- 一个下一步
- 一次重新聚焦

而不是一份冗长总结。

### `5. AI Providers`

这是模型接入层。

它负责：

- `OpenAI`
- `Ollama`
- `None`

它应该可插拔、可切换。

## 5. The Correct User Flow

正确的产品路径应该是：

```txt
用户看到桌面上的团子
  -> 点一下她 / 直接开口
  -> 团子先回应
  -> 判断这是情绪、闲聊、还是任务
  -> 如果有必要，再记成主线 / 阻塞 / 下一步
  -> 后续按节奏轻提醒
  -> 在卡住或分心时帮助重新启动
```

重点在这里：

- `回复` 在前
- `记忆` 在后

## 6. What We Must Not Build

下面这些东西现在都不应该成为主线：

### `复杂任务面板优先`

如果界面第一眼像：

- 看板
- 表格
- 项目面板

那就偏了。

### `把项目认知做成项目管理器`

后台识别“这批事属于哪条项目线”是对的。

但不能一路滑成：

- 项目总览页
- 多列项目卡片
- 依赖关系图
- 大型 roadmap

否则又会把“任务搭子”推回“复杂效率工具”。

### `周报/日报生成器`

梳理能力可以有，但它只是辅助。

不能把产品核心变成：

- 总结你今天做了什么
- 自动帮你出周报

这会把产品从“任务搭子”拉偏成“汇报助手”。

### `大而全效率工具箱`

不应该一上来塞满：

- 日历
- 笔记
- 多层项目
- 数据统计页
- 大型仪表盘

### `没有人格的 AI 聊天壳`

如果只是把普通聊天框挂到桌面上，也是不对的。

必须保证：

- 她有持续身份
- 她有情绪和语气一致性
- 她知道你现在手里的主线

## 7. MVP We Actually Want

第一版真正应该做的 MVP 只有这些：

1. 一个桌面常驻小人
2. 点击后能直接聊天
3. 她有稳定人格和语气
4. 能把一句话识别成 `情绪 / 任务 / 阻塞 / 完成`
5. 只记住少量关键任务记忆
6. 有轻提醒和轻梳理
7. 支持 `Ollama / OpenAI`

## 8. Current Drift

当前版本已经出现过的跑偏方向：

- 把任务板做得太显眼
- 让“梳理”看起来像周报
- 把用户交互做成了“填任务”
- 桌宠存在感还不够强

这些都不是实现失败，而是结构还没锁死前自然会发生的偏航。

从现在开始，判断一项功能是否应该做，只看一个问题：

**它是在加强“桌面上的任务搭子”，还是在把产品推向“另一个效率工具”？**

## 9. Near-Term Execution Order

后续继续开发时，顺序应该是：

1. 稳定 `Pet Shell` 的存在感
2. 稳定 `Conversation Core` 的人格和回复体验
3. 把 `Task Memory` 收缩成真正的轻记忆
4. 把 `Guidance Engine` 做成轻提醒，不做报表
5. 最后再做 provider 切换、行为感知和更复杂扩展

## 10. One-Sentence Check

以后每次要加功能前，先问自己：

**这个功能会不会让团子更像一个在桌面上陪我启动行动的搭子？**

如果答案不是明确的“会”，就先别做。
