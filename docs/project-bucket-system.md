# Project Bucket System

这份文档回答一个很具体的问题：

**团子到底应该用几个“项目桶”来理解你手上的事？**

结论先写在前面：

- 不建议只用一层粗桶，比如“文娱 / 研究 / 代码”
- 更稳的是 **两层结构**
- 第一层决定它 **怎么想、怎么打分、该查什么证据**
- 第二层决定它 **这条线具体属于哪个主题**

## 1. Why One Layer Is Not Enough

如果只做一层桶，很容易把两种完全不同的东西混在一起：

- `研究 / 代码 / 申请`
  - 这是 **工作方式**
- `EEG / 翻译 / 文娱 / 申请`
  - 这是 **主题语境**

比如 `EEG`：

- 它不是纯研究
- 也不是纯代码
- 更像 **产品 / 设计 / 人因 / 硬件接触问题**

所以桶系统不能只看“主题”，也不能只看“文件类型”。

## 2. Recommended Structure

### Layer 1: Thinking Buckets

这一层决定：

- 该优先看什么证据
- 风险怎么打
- 需不需要外搜
- 它应该怎么主动检查进度

我建议第一版用 `6` 个主桶：

#### `1. 代码实现类`

适合：

- repo / app / plugin / backend / frontend / shell
- 目标是把东西做出来、修掉、连起来

它应优先看：

- workspace / repo 结构
- 入口文件
- 最近线程标题
- build / issue / next action

典型问题：

- 入口在哪
- 这条实现线卡在哪
- 现在是架构问题还是修 bug

#### `2. 研究评估类`

适合：

- benchmark
- 实验
- 指标
- baseline
- 方法验证

它应优先看：

- 数据集 / benchmark / eval 脚本
- 指标定义
- baseline / ablation / result
- 外部论文或公开标准

典型问题：

- 这个实验到底在比什么
- 当前结果可信不可信
- 应该优先补数据、补评测，还是补论文对齐

#### `3. 产品 / 设计类`

适合：

- 产品概念
- 交互方案
- 机械结构
- 人因 / 佩戴 / 接触几何
- 竞品比较

它应优先看：

- 用户问题
- 使用场景
- 接触路径 / 结构路径
- 竞品能力
- 设计 tradeoff

典型问题：

- 这个方案是不是更顺手
- 结构是不是能自然落位
- 它是在解决真实用户问题，还是只是技术上可做

#### `4. 申请 / 写作类`

适合：

- PhD application
- SOP / proposal / CV
- 推荐信
- 学校定制材料
- 提交与版本运营

它应优先看：

- 材料成熟度
- 截止时间
- 版本差异
- claim 与证据边界
- 提交状态

典型问题：

- 哪个学校版本还没成型
- 哪条叙事该主打
- 哪个 claim 还站不住

#### `5. 运营 / 事务类`

适合：

- 文件搬运
- 清理
- 汇总
- 导出
- 报告包装
- 日程型推进

它应优先看：

- 清单
- 依赖
- 状态
- 完成条件

典型问题：

- 现在该先清哪块
- 哪些已经完成
- 哪些还差一步就能交付

#### `6. 生活 / 文娱类`

适合：

- 轻提醒
- 兴趣内容
- 放松
- 生活偏好

它应优先看：

- 节奏
- 偏好
- 最近状态

典型问题：

- 现在适合做什么轻量的事
- 想放松时该推什么

### Layer 2: Topic Tags

这一层不决定评分逻辑，只决定主题归属。

比如：

- `桌宠`
- `EEG`
- `翻译`
- `申请`
- `视觉`
- `磁盘清理`
- `报告`

所以一个项目应该是：

- `研究评估类 + 翻译`
- `产品 / 设计类 + EEG`
- `申请 / 写作类 + PhD application`
- `代码实现类 + 桌宠`

而不是只有一个扁平大类。

## 5. How The Buckets Should Be Used

这些桶不是给 UI 好看用的，它们应该真正改变团子的行为。

### 对 `代码实现类`

它应优先：

- 看 repo skeleton
- 看入口
- 看最近线程
- 看 next step

而不是先去搜一堆泛网页。

### 对 `研究评估类`

它应优先：

- 看 benchmark / eval evidence
- 看 metric / baseline / result
- 必要时查外部论文或公开标准

### 对 `产品 / 设计类`

它应优先：

- 看设计问题是否明确
- 看结构与接触路径
- 看竞品与用户问题

### 对 `申请 / 写作类`

它应优先：

- 看材料成熟度
- 看版本与学校差异
- 看 deadline / submission / evidence safety

### 对 `运营 / 事务类`

它应优先：

- 看清单和状态
- 不要过度“研究脑”

### 对 `生活 / 文娱类`

它应优先：

- 轻交互
- 低压力
- 不要误上重型项目监督

## 6. Practical Rule

以后如果要新增桶，先过这两个问题：

1. 它是不是在改变“团子该怎么想、怎么评分、怎么查证据”？
2. 它是不是只是一个主题标签，而不是新主桶？

只有第一个答案是 `是`，才值得新增主桶。

## 7. Immediate Recommendation

如果按“现在立刻能落地”的顺序，我建议是：

1. 保留当前已落地的 6 个主桶：
   - `engineering_execution`
   - `research_evaluation`
   - `product_research`
   - `application_ops`
   - `operations_admin`
   - `life_entertainment`
2. 把 UI 文案改成人话：
   - `代码实现类`
   - `研究评估类`
   - `产品 / 设计类`
   - `申请 / 写作类`
   - `运营 / 事务类`
   - `生活 / 文娱类`
   - `暂未定类`
3. 下一阶段把每个桶的评分依据做成可见 evidence panel，而不是只显示一个标签

这样不会一下把现有代码打散，但产品语言已经会顺很多。
