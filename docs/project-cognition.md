# Project Cognition

这份文档说明一个新能力：

**当用户把一大串混合事项丢给团子时，她应该怎么理解。**

这不是让产品转向“大而全项目管理”，而是给桌面搭子补一层后台认知。

## 1. Why It Exists

用户真实会说的话，常常不是：

- “新增任务：重做 pitch”

而是：

- “做 report 衡量各个技术标准”
- “光学模组专利申请”
- “样品生产”
- “总结一下我们商业东西都做了什么”

这类输入里混着：

- 项目
- 子事项
- 商业动作
- 研发动作
- 待补梳理

如果没有项目认知层，系统只能把它们误当成：

- 一堆散任务
- 或一段普通聊天

这就是之前会跑成“周报梳理器”的原因之一。

## 2. What It Actually Is

`Project Cognition` 是一层**后台项目线识别与归并能力**。

它的职责是：

1. 识别用户这次发来的是否是“混合清单 / 项目 dump”
2. 把混合事项归并成少量项目线
3. 判断哪些像旧项目，哪些像新项目
4. 给前台对话输出一段简短的人话总结
5. 把这些项目线以轻量记忆的方式存起来

它不是：

- 项目管理面板
- 甘特图
- 多级任务树
- 周报生成器

## 3. Relationship To Final Product Structure

它不单独升级成新的顶层产品形态。

它属于原来 `Task Memory` 的内部拆分：

- `Active Memory`
  - 当前主线
  - 阻塞点
  - 下一步
  - 提醒节奏
- `Project Cognition`
  - 这批事项像哪条项目线
  - 是旧项目还是新项目
  - 每条项目线最近挂着什么

所以前台看起来仍然是：

- 小人
- 对话
- 少量记忆卡片

而不是一个项目管理软件。

## 4. Correct User Flow

正确路径应该是：

```txt
用户发来一串混合事项
  -> 团子先判断这是不是 backlog dump
  -> 如果是，就先在后台做项目归并
  -> 得到 3 到 6 条项目线
  -> 用 2 到 3 句话回用户
  -> 再把项目线以轻记忆方式存起来
```

前台回复重点是：

- “我看出来这是几条项目线”
- “哪些像旧项目”
- “下一步先补哪条”

而不是：

- “以下是你今天的 11 个待办事项”

## 5. MVP Output

第一版只需要输出这些：

- 一句摘要
- 3 到 5 条项目线名称
- 每条项目线下面挂着的事项
- 一句下一步建议

第一版先不做：

- 项目 owner
- 项目 timeline
- 跨项目依赖图
- 项目状态大盘

## 6. Storage

项目认知层用独立本地文件存储：

- `projects.json`

每条项目线只保留轻量信息：

- `Name`
- `Summary`
- `KindLabel`
- `Keywords`
- `RecentItems`
- `UpdatedAt`

这意味着它更像“记住你最近在推进哪些线”，
而不是“建立一个企业项目数据库”。

## 7. Product Boundary

以后如果要判断某个需求该不该做，先问：

**它是在帮助团子更好地认出“这堆事属于哪条项目线”，还是在把团子推向项目管理器？**

如果更像后者，就先别做。
